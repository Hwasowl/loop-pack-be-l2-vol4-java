# Round 5 읽기 최적화 — 의사결정 맥락

> 밥 먹고 온 사이 진행된 작업의 **왜**를 정리한 문서. 무엇을 했는지(코드/수치)는 [03_benchmark_report.md](03_benchmark_report.md)에 있고, 여기는 **판단의 근거와 트레이드오프**만 담는다.

## TL;DR

- 출발점은 **① 인덱스**로 잡았다. 캐시(③)는 인덱스 없는 쿼리 위에 바르는 반창고라, 캐시 미스 한 번이 곧 풀스캔이 되어 최악의 순간에 무너진다. 기저 쿼리를 먼저 빠르게 만든다.
- 과제 ②(좋아요 수 정렬 구조)는 **이미 완료** 상태였다 — `product.like_count` 비정규화 컬럼 + 원자적 `UPDATE like_count = like_count ± 1` 동기화. 새로 만들 것 없음.
- 상품 10만건을 시딩하고 `EXPLAIN ANALYZE`로 **AS-IS(전부 풀스캔 + filesort)** 를 재현한 뒤, 실행계획을 보고 인덱스를 설계했다. **추측으로 인덱스를 박지 않았다.**
- 두 가지 설계 결정을 내렸다(아래). 둘 다 측정 결과를 근거로 했고, 하나는 기존 코드를 건드리는 사안이라 확인을 받았다.

## 출발점을 ① 인덱스로 잡은 이유

시니어 아키텍트 관점 분석에서 "가장 먼저 손대야 할 위험 요소"가 **인덱스 부재**로 나왔다. 근거:

- 현재 모든 상품 목록 조회는 `brand_id` 필터 + `ORDER BY`를 **테이블 풀스캔 + filesort**로 처리한다(인덱스 PK 외 전무). 10만건에서 매 요청이 10만 행을 훑는다.
- 캐시를 먼저 얹어도, 캐시 미스·TTL 만료·thundering herd 순간에 그대로 풀스캔이 DB로 꽂힌다. 즉 **캐시는 인덱스 위에 얹어야 안전하다.**
- 반대로 가장 나중까지 미뤄도 되는 건 Materialized View / Pre-aggregation 이다 — `like_count` 비정규화가 이미 그 역할을 하고 있어 지금 도입 가치가 가장 낮다.

순서: **시딩 → EXPLAIN(AS-IS) → 인덱스 설계·적용 → EXPLAIN(TO-BE) → 리포트 → (다음) 캐시.**

## 결정 1 — tiebreak 방향을 정렬 방향에 맞춘다

**변경**: `ProductRepositoryImpl.toSort()`의 `PRICE_ASC` tiebreak을 `id DESC` → `id ASC`로 바꿨다.
(`LATEST`/`LIKES_DESC`는 주 정렬이 DESC라 `id DESC`가 이미 방향 일치 → 그대로 둠.)

**왜 — 측정으로 드러난 사실:**

페이지 경계 안정성을 위해 모든 정렬에 `id` tiebreak을 붙인다. 그런데 tiebreak 방향이 주 정렬 키와 **엇갈리면** 복합 인덱스가 정렬을 끝까지 채우지 못한다.

| 정렬 | tiebreak | 인덱스 정렬 커버 | filesort |
|---|---|---|---|
| `price ASC` | `id DESC` (엇갈림) | ❌ 못 채움 | 잔존 (입력만 980행으로 축소) |
| `price ASC` | `id ASC` (일치) | ✅ ordered 스캔 | **제거** |
| `like_count DESC` | `id DESC` (일치) | ✅ reverse 스캔 | 제거 |

- InnoDB 보조 인덱스는 끝에 PK(`id`)가 **오름차순**으로 붙는다. 그래서 `(brand_id, price)` 인덱스는 `price ASC, id ASC` 순서를 그대로 제공한다 → `id ASC` tiebreak이면 정렬 연산 없이 인덱스만으로 끝난다.
- `id DESC` tiebreak이면 방향이 엇갈려 옵티마이저가 정렬을 다시 한다(작은 filesort). 실제로 옵티마이저는 이때 `idx_product_brand_price`를 외면하고 다른 brand 인덱스를 골랐다 → **`(brand_id, price)` 인덱스가 read 이득 없이 write 비용만 내는 잉여**가 될 뻔했다.

**트레이드오프 / 안전성:**

- 사용자 체감 변화 거의 없음 — `price`는 거의 unique(10만건에 1천~200만 분포)라 동일 가격 ties가 드물고, ties일 때만 id 순서가 바뀐다.
- 페이지네이션 안정성은 그대로 유지된다(여전히 결정적 tiebreak). 방향만 정렬 키에 맞춘 것.
- "정렬 방향에 맞춘 tiebreak"이 규칙이 됐으므로, 향후 정렬 옵션이 추가되면 같은 원칙을 적용한다.

## 결정 2 — 인덱스 범위: brand 복합 3 + 전역 인기 1

**적용한 인덱스** (`ProductModel`의 `@Table(indexes=...)`):

| 인덱스 | 컬럼 | 커버하는 조회 |
|---|---|---|
| `idx_product_brand_like` | `(brand_id, like_count)` | 브랜드 필터 + 인기순 |
| `idx_product_brand_price` | `(brand_id, price)` | 브랜드 필터 + 가격순 |
| `idx_product_brand_created` | `(brand_id, created_at)` | 브랜드 필터 + 최신순 |
| `idx_product_like` | `(like_count)` | **전역** 인기순 (헤드라인 유스케이스) |

**왜 6개가 아니라 4개인가 (전역 price/최신 인덱스 제외):**

- `(brand_id, X)` 복합 인덱스는 **전역(브랜드 미지정) 정렬에는 못 쓴다** — 전역 쿼리엔 `brand_id` 등치 조건이 없어 인덱스 선두 컬럼을 못 타기 때문. 전역 각 정렬을 커버하려면 `(price)`, `(created_at)` 단일 인덱스를 따로 만들어야 한다(총 6개).
- 인덱스를 6개로 늘리면 `product` 테이블의 모든 write — 상품 등록, 그리고 **좋아요 누를 때마다 일어나는 `like_count` UPDATE** — 가 인덱스 6개를 갱신한다. 좋아요는 빈번한 쓰기라 비용이 누적된다.
- Round 5의 헤드라인은 "**인기 상품 조회**"다. 전역 인기순(`idx_product_like`)은 넣고, **전역 가격순·최신순은 후순위로 미뤘다**(분석의 Q5 — "가장 나중까지 미룰 수 있는 개선"과 일치). 실제 트래픽에서 전역 가격/최신 정렬 수요가 관측되면 그때 단일 인덱스를 추가한다.

**의도적으로 남긴 갭 (silent 처리 아님):**

1. **전역 `price ASC` / `latest` 정렬** → 인덱스 없음, 여전히 풀스캔. 수요 관측 시 `(price)`, `(created_at)` 추가.
2. **깊은 OFFSET 페이지네이션** → 인덱스가 있어도 `OFFSET 50000`은 5만 행을 읽고 버려서 비싸다(측정: 인덱스 무관하게 느림). 이건 인덱스 문제가 아니라 **OFFSET → 커서 기반 페이지네이션** 전환으로 풀어야 하는 별도 과제 → 이번 범위 밖.

## 다음 단계 (③ 캐시)

인덱스로 기저 쿼리를 빠르게 만들었으니, 그 위에 Redis 캐시를 얹는다. 순서는 **상세(단건·키 단순·재요청 집중 → 적중률 높음) 먼저, 목록(필터×정렬×페이지 조합으로 키 폭발 → 적중률 낮음)은 후순위.** TTL + 무효화 전략, 캐시 미스 시 정상 동작까지 포함한다.
